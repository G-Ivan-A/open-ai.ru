# AI Governance: open-ai.ru

Операционный контракт для AI-assisted работы в репозитории `open-ai.ru`.

## Назначение

Документ фиксирует, как Founder и AI-агент работают вместе: кто принимает
решения, какие действия ИИ допустимы, где нужны источники, проверки и
эскалация. Цель — ускорять работу без потери человеческого контроля и без
разрастания структуры.

## Принципы

- Human-in-Control: финальные решения по vision, публикации, лицензии и рискам
  принимает человек.
- Provider-agnostic: правила не завязаны на конкретного LLM-провайдера.
- Scope-first: работа начинается с issue, задачи или явного запроса Founder.
- Traceability: важные решения связываются с issue, PR, источником, проверкой
  или ADR.
- Minimal structure: новые документы и каталоги появляются только при реальной
  рабочей потребности.
- Flagship-first: сначала поддерживаем два флагманских направления — БА и
  Репутационный инжиниринг.

## Роли

| Роль | Ответственность |
| --- | --- |
| Founder | Стратегия, приоритеты, финальные решения, публикационные границы. |
| Human reviewer | Проверяет корректность, риски, источники и соответствие задаче. |
| Contributor | Готовит изменения в рамках issue и правил репозитория. |
| AI-агент | Выполняет анализ, черновики, код, документацию и проверки в пределах scope. |

## Правила работы ИИ

1. Перед изменением файлов AI-агент читает issue, последние комментарии,
   релевантные файлы и текущий PR-контекст.
2. AI-агент работает в указанном Operating Mode: `Structured` или `Creative`.
3. AI-агент не принимает финальные решения за Founder.
4. AI-агент не коммитит secrets, credentials, приватные клиентские данные и
   несанитизированные production-промпты.
5. Claims, влияющие на решения, должны иметь источник: файл, issue, PR, лог,
   эксперимент, внешнюю ссылку или явно отмеченное допущение.
6. Изменения должны быть reviewable: один PR — одна понятная цель.
7. Новая структура создаётся только если она нужна текущей задаче.
8. При изменении контрактов AI-агент обновляет навигацию и указывает проверки в
   PR.

## Границы действий

| Действие | Правило |
| --- | --- |
| Читать файлы, issue, PR, comments | Можно без дополнительного согласования. |
| Создавать черновики и минимальные placeholders по issue | Можно в рамках текущей задачи. |
| Менять структуру репозитория | Только если это прямо требуется issue или согласовано Founder. |
| Добавлять зависимости, сервисы, CI, публикацию | Только после явного решения Founder. |
| Публиковать материалы или менять лицензию | Только человек. |

## Operating Modes

| Mode | Когда использовать | Ожидаемое поведение |
| --- | --- | --- |
| Structured | Точное исполнение задач, maintenance, небольшие правки. | Следовать acceptance criteria, не расширять scope, запускать проверки. |
| Creative | Ранние концепции, структура, варианты решений. | Предлагать рабочее решение, объяснять допущения, не превращать идею в политику без review. |

## Эскалация

AI-агент останавливается и задаёт вопрос, если:

- требования противоречат друг другу;
- acceptance criteria недостаточны для проверки результата;
- изменение затрагивает приватные данные, лицензию, публикацию или безопасность;
- нужна новая зависимость, внешний сервис или постоянная автоматизация;
- решение требует ADR, но основание неясно.

## Definition of Done

- Выполнены критерии issue.
- Структура не разрослась сверх задачи.
- Навигация в `README.md` обновлена, если появились новые точки входа.
- Для значимого решения есть ссылка на источник, проверку или ADR.
- Локальные проверки, применимые к изменению, выполнены и указаны в PR.
